# Documentation Index

## 1. Purpose

This document serves as the master index for the T.N. Memorial Public School Digital Platform documentation suite. It provides a structured entry point for navigating the repository’s SDLC documentation and supporting materials.

## 2. Documentation Architecture

The documentation is organized by lifecycle domain and follows a consistent numbering and naming structure. Each folder contains documentation relevant to a specific phase of delivery, governance, or support.

## 3. Folder Descriptions

- 00_Project_Governance: repository governance, project oversight, and foundational documentation.
- 01_Discovery: vision, scope, stakeholder, and current-state analysis.
- 02_Business: business requirements, rules, and process flow documentation.
- 03_Requirements: functional requirements and scope baseline.
- 04_Architecture: system and component architecture guidance.
- 05_Database: database design, schema, tables, and data dictionary.
- 06_API: REST API standards and endpoint documentation.
- 07_UI: UI design standards and interface guidance.
- 08_Security: security principles, controls, and review practices.
- 09_Testing: testing strategy and validation guidance.
- 10_Deployment: deployment, operations, and release readiness guidance.
- 11_Portfolio: portfolio-oriented project documentation.
- 12_ADR: architecture decision records.
- 13_RTM: requirements traceability matrix.
- 14_QMS: quality management system documentation.
- 15_Appendix: supporting reference material and navigation aids.
- 16_Architecture_Handbook: architecture guidance and implementation handbook.
- 17_AI_Prompts: prompt templates and AI-assisted workflow guidance.

## 4. SDLC Phases

The documentation suite supports the following phases:

- Discovery
- Business Analysis
- Requirements Definition
- Architecture Design
- Database Design
- API Design
- UI Design
- Security Design
- Testing Planning
- Deployment and Operations

## 6. Document Numbering Rules

Documentation files follow the repository’s numbering and naming conventions, including prefixes such as ARC, API, DB, UI, SEC, TST, DEP, PRT, ADR, RTM, QMS, APP, and AIP.

## 7. Repository Documentation Workflow

The documentation workflow follows a structured sequence:

1. Capture requirements and scope.
2. Define architecture and data structures.
3. Document interfaces and experience standards.
4. Define security and quality controls.
5. Plan testing and deployment.
6. Preserve traceability and operational guidance.

Cross references should be used to connect related documentation across requirements, architecture, API, database, UI, security, testing, and deployment domains.
